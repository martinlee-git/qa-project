## Original Description

Redis CVE-2026-23479 enables authenticated RCE; affecting versions since 7.2.0, patched May 5 to reduce exploitation risk.
## Key Points
- Redis CVE-2026-23479는 인증된 RCE를 활성화합니다. 7.2.0 이후 버전에 영향을 미치며 악용 위험을 줄이기 위해 5월 5일에 패치가 적용되었습니다.
- CVE-2026-23479로 추적되는 이 결함은 Redis 7.2.0에 도입되었으며 5월 5일 수정 전까지 모든 안정적인 브랜치에 남아 있었으며 2년 넘게 눈에 띄지 않았습니다.
- NVD는 CVSS 3.1에서 8.8을 평가했습니다. Redis는 CVSS 4.0에서 7.7로 나열합니다.
## Excerpt
CVE-2026-23479로 추적되는 이 결함은 Redis 7.2.0에 도입되었으며 5월 5일 수정 전까지 모든 안정적인 브랜치에 남아 있었으며 2년 넘게 눈에 띄지 않았습니다. NVD는 CVSS 3.1에서 8.8을 평가했습니다. Redis는 CVSS 4.0에서 7.7로 나열합니다. 이는 Team Xint Code에 의해 보고되었으며 이제 완전한 기술 문서가 공개되었습니다. 클라우드 공간이 상황을 더욱 악화시킵니다. 익스플로잇 글과 함께 게시된 Wiz의 분석에 따르면 Redis는 대부분의 클라우드 환경에 있으며 대부분의 인스턴스는 비밀번호 없이 실행됩니다. 익스플로잇에는 인증된 세션이 필요하지만 기본 배포에서는 기본 사용자가 이미 체인에 필요한 모든 권한을 보유하고 있습니다. 이 결함은 src/blocked.c의 unblockClientOnKey()에 존재하며, 이는 키 이벤트가 차단된 명령을 깨울 때 발생합니다. 함수는 processCommandAndResetClient() 를 통해 대기 중인 명령을 전달한 다음 동일한 클라이언트 포인터를 계속 사용합니다. 문제: 해당 함수는 부작용으로 클라이언트를 해제할 수 있으며 자체 헤더 주석에 그렇게 나와 있습니다. 호출자는 반환 값을 무시하고 해제 후 사용(CWE-416) 방식으로 해제된 구조를 읽습니다. Wiz의 분석에 따르면 버그를 생성하는 데 두 번의 커밋이 필요했습니다. 2023년 1월 리팩터링(PR #11012)에서 확인되지 않은 호출을 추가했습니다. 2023년 3월 변경사항(PR #11568)으로 인해 이후에 더 많은 클라이언트 액세스가 추가되었습니다. 혼자서는 위험하지도 않았습니다. 함께 7.2.0에서 일반 가용성에 도달했으며 여러 차례의 보안 검토를 통과했습니다. 체인은 힙 주소 유출로 시작됩니다. 거기에서 클라이언트를 해방시키고 FA를 미끄러뜨립니다.하나를 동일한 메모리에 넣은 다음 Redis 자체 메모리 계정을 자체적으로 변경하여 함수 포인터를 덮어씁니다.
